# 訂單流程與狀態機稽核報告 v1.0.1

## 已確認並修正的問題

1. 付款完成後只把購物車標示為 `CONVERTED`，但仍保留品項，回首頁後仍顯示可付款購物車。
2. 成功頁的「查看訂單」沒有實際路由與訂單資料。
3. 結帳頁在購物車與房間皆空時會自動塞入示範餐點，造成已付款後可再次付款。
4. 付款成功頁沒有真實訂單時仍顯示固定成功資訊。
5. 付款中重新整理後沒有中斷恢復規則。
6. 一起點付款後活動房間仍保留，可再次形成結帳內容。
7. 房間倒數只更新畫面，沒有把 Store 狀態從 `OPEN` 切換為 `LOCKED`。
8. 直接開啟房間網址會自動建立示範房間，繞過建立／加入流程。
9. 沒有已提交餐點的房間仍可進入結帳。
10. 顧客付款後的訂單沒有進入管理端，管理端只有固定示範資料。
11. 管理端接單與顧客訂單狀態沒有共用同一個訂單狀態機。

## 修正後閉環

### 一般訂單

`EMPTY → EDITING → CHECKOUT_SUBMITTING → PAID → ACCEPTED → PREPARING → READY_FOR_PICKUP／OUT_FOR_DELIVERY → COMPLETED`

付款成功時會在同一個 Store 動作中：

- 建立不可變訂單快照。
- 保存餐點、選項、金額、分店、模式、付款 ID 與建立時間。
- 清空活動購物車與折扣碼。
- 關閉活動一起點房間。
- 清除付款中的嘗試識別。
- 保存 `lastOrderId` 供成功頁、首頁與結帳守衛使用。

### 一起點房間

`OPEN → LOCKED → REVIEWING → CHECKOUT_PENDING → PAYMENT_PENDING → 訂單 PAID → 房間清除`

規則：

- 倒數到期時真正執行 `OPEN → LOCKED`。
- `LOCKED` 不接受新的參與者提交。
- 延長時間可合法 `LOCKED／REVIEWING／CHECKOUT_PENDING → OPEN`。
- 沒有已提交餐點時禁止進入結帳。
- 付款完成後參與者與品項保存至訂單快照，活動房間清除。

### 付款中斷

- 持久化資料若顯示 `PAYMENT_PENDING`，重新載入時轉成 `FAILED`。
- 不會假裝仍在付款，也不會自動標示成功。
- 使用者可保留有效購物車並重新發起付款。

### 路由守衛

- `/order/success` 沒有真實訂單時顯示「找不到訂單」。
- `/checkout` 沒有有效餐點時導向最近訂單或購物車。
- 已付款後返回 `/checkout` 不顯示付款按鈕。
- `/group/room` 沒有活動房間時導向 `/group/join`。

## 持久化

- 新版本 key：`dongji-order-state-v2`。
- 舊版本 `dongji-order-state-v1` 自動遷移。
- 舊版 `CONVERTED` 購物車遷移後直接清空。
- 訂單完成事件具冪等性，相同訂單 ID 重送不重複建立。

## Demo 版本限制

目前持久化採瀏覽器 localStorage，管理端與顧客端必須在同一瀏覽器來源才能共享訂單。正式多裝置、多分店營運必須改接伺服器資料庫與即時服務；這是 Infrastructure 層替換，不需改動 Domain 狀態機。
